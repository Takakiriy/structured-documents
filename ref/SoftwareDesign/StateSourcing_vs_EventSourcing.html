詳細な比較

State Sourcing

メリット:
  • 実装が最もシンプル
  • 読み書き共に最速
  • ストレージ最小
  • 標準的なCRUD
デメリット:
  • 履歴が一切残らない
  • 監査証跡なし
  • デバッグ困難
  • 復元不可

Temporal State（中間アプローチ）

メリット:
  • 実装が比較的簡単
  • 読み書き高速
  • いつ変化したかが分かる
  • ソフトデリートが容易
  • コンプライアンスに有効
デメリット:
  • 何が変化したかは不明
  • 完全な履歴は残らない
  • 細かい追跡は困難
  • 別途履歴テーブル必要な場合も

Event Sourcing

メリット:
  • 完全な履歴・監査証跡
  • 任意の時点に復元可能
  • タイムトラベルデバッグ
  • 豊富なビジネスインサイト
  • イベント駆動と相性◎
デメリット:
  • 実装が複雑
  • 学習コスト高
  • 読み取りに再生が必要
  • ストレージ容量大
  • スキーマバージョニング必要

選択基準

State Sourcingを選ぶ場合

  • 履歴が不要
  • 現在の状態だけが重要
  • シンプルさ最優先
  • パフォーマンス最優先
  • 小規模プロジェクト
  • プロトタイプ開発

Temporal Stateを選ぶ場合

  • いつ変化したかが重要
  • ソフトデリート必要
  • 基本的な監査要件
  • 実装の簡単さも重視
  • 中規模プロジェクト
  • 一般的なWebアプリ

Event Sourcingを選ぶ場合

  • 完全な履歴が必須
  • 監査証跡が重要
  • コンプライアンス要件
  • 複雑なビジネスロジック
  • データ分析が重要
  • イベント駆動設計

ハイブリッドアプローチ

実際のシステムでは、複数のアプローチを組み合わせることも一般的です：

パターン1: Event Sourcing + スナップショット

イベントは全て記録しつつ、パフォーマンスのために定期的にスナップショット（State）を保存。 読み取り時は最新スナップショット + 以降のイベントで再生。

パターン2: Temporal State + 監査ログ

メインテーブルは Temporal State で管理し、重要な操作だけ別途監査ログテーブルに記録。 バランスの取れたアプローチ。

パターン3: 機能別に使い分け

重要な取引データは Event Sourcing、ユーザープロフィールは Temporal State、 設定値は State Sourcing というように、データの性質に応じて使い分ける。

💡 重要なポイント

State Sourcing: 「写真」- その瞬間だけを記録

Temporal State: 「写真集」- いつ撮影されたかも記録（中間アプローチ）

Event Sourcing: 「動画」- 全ての変化を記録

Temporal State は実務で最もよく使われるバランスの良いアプローチです。 created_at、updated_at、deleted_at などのタイムスタンプを持つことで、 実装の複雑さを抑えつつ、基本的な監査要件を満たすことができます。
