状態管理・イミュータビリティの概念マップ

共通思想: イミュータビリティ（不変性）
予測可能性
同じ入力→同じ出力
追跡可能性
変更履歴を保持
Read-Only
追記のみ、変更不可
▶ Redux - アプリケーション状態管理
3つの原則
  • • Single Source of Truth（単一Store）
  • • State is Read-only（Actionで変更）
  • • Pure Functions（Reducerで更新）
データフロー
Action → Reducer → New State → View
類似ライブラリ
Zustand（軽量）
Jotai（Atomic）
Recoil（Meta製）
MobX（Observable）
▶ イミュータブル更新ライブラリ
Immer ⭐
変更風の書き方→新しいオブジェクト生成
produce(state, draft => { draft.user.name = 'John'; })
※ Redux Toolkitに標準搭載
その他
Immutable.js（永続的データ構造）
seamless-immutable（軽量）
▶ Event Sourcing - イベント駆動設計
基本概念
状態ではなく「変更イベント」を時系列で保存
イベントログの例
1. { type: 'ACCOUNT_CREATED', amount: 0 } 2. { type: 'DEPOSITED', amount: 100 } 3. { type: 'WITHDRAWN', amount: 50 } 4. { type: 'DEPOSITED', amount: 50 } → 現在の残高: 100円（イベントを再生して計算）
メリット
  • • 完全な履歴を保持（監査証跡）
  • • 任意の時点に状態を復元可能
  • • タイムトラベルデバッグ
関連技術
CQRS（Command Query Responsibility Segregation）
▶ Append-only Log - 分散システム基盤
Append-only の原則
追記のみ、過去のエントリは変更・削除不可
主な実装
Apache Kafka
分散イベントストリーミング、Commit log
Write-Ahead Log (WAL)
PostgreSQL、Redisなど
分散トレーシング
マイクロサービス間のリクエスト追跡
Jaeger
Zipkin
OpenTelemetry
Spanを時系列に記録し、処理フローを可視化
概念間の関係
Redux + Immer = Redux Toolkit（公式推奨）
Event Sourcing → Redux のAction履歴に影響
Append-only Log → Event Sourcing の基盤技術
